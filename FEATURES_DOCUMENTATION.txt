--------------------------------------------------------------------------------
1. DASHBOARD MEETING RESPONSE SYSTEM
--------------------------------------------------------------------------------

Description: Participants can respond to meeting invitations directly from 
the dashboard with Accept, Maybe, or Decline options.

Features:
- Three colored response buttons (Green Accept, Amber Maybe, Red Decline)
- Real-time UI updates showing current response status
- "Pending Invites" counter updates automatically
- Only visible to participants (not organizers)
- Hidden for completed meetings
- Label changes from "Respond:" to "Change:" after initial response

Location: Dashboard page → Upcoming Meetings section

Permissions:
- Participants: Can respond to their own invitations
- Organizers: Cannot respond to meetings they organize

Technical Details:
- Frontend: /app/frontend/src/pages/DashboardPage.js
- Backend API: PUT /api/meetings/{meeting_id}/participants/{user_id}/response
- Database: meeting_participants collection (response_status field)

--------------------------------------------------------------------------------
2. 7-DAY TREATMENT PLAN EDIT WINDOW
--------------------------------------------------------------------------------

Description: After a meeting is marked as "completed", treatment plans can 
be edited for 7 days, then become read-only.

Features:
- Automatic timestamp recording when meeting status changes to "completed"
- Real-time countdown showing remaining edit days
- Warning banner: "Treatment plan can be edited for X more days"
- After 7 days: Red alert "The 7-day edit window has expired"
- Edit button disabled after expiration
- Both organizers and participants can edit within the window

Location: Meeting Detail Page → Agenda Tab → Treatment Plan section

Permissions:
- Organizers: Can edit within 7-day window
- Participants: Can edit within 7-day window
- Everyone: Read-only after 7 days

Technical Details:
- Frontend: /app/frontend/src/pages/MeetingDetailPage.js
  - Helper functions: isTreatmentPlanEditable(), getRemainingEditDays()
- Backend API: PUT /api/meetings/{meeting_id}/agenda/{item_id}/treatment-plan
- Database: meetings collection (completed_at timestamp field)

--------------------------------------------------------------------------------
3. PARTICIPANTS CAN ADD PARTICIPANTS TO MEETINGS
--------------------------------------------------------------------------------

Description: Any meeting participant (not just organizers) can invite other 
participants to the meeting.

Features:
- "Add Participant" button visible to all participants
- Two tabs: "Existing Doctors" and "Invite by Email"
- Add existing users from system
- Invite new users by email (creates account automatically)
- Default password for new users: TempPass123!
- Backend permission validation
- Tracking of who added each participant (added_by field)

Location: Meeting Detail Page → Overview Tab → Participants Section

Permissions:
- Organizers: Can add participants
- Participants: Can add participants
- Non-members: Cannot add participants

Technical Details:
- Frontend: /app/frontend/src/pages/MeetingDetailPage.js
- Backend API: POST /api/meetings/{meeting_id}/participants
- Database: meeting_participants collection

--------------------------------------------------------------------------------
4. EMAIL INVITE BUG FIX
--------------------------------------------------------------------------------

Description: Fixed "body stream already read" error when inviting new 
participants via email.

Issue: Frontend was sending "full_name" but backend expected "name" field, 
causing validation failure and response body re-reading error.

Fix Applied:
- Changed frontend to send "name" instead of "full_name"
- Improved error handling to prevent response body re-reading
- Added JSON parsing error handling
- Better error messages for all failure scenarios

Location: Meeting Detail Page → Participants Tab → Invite by Email

Technical Details:
- Frontend: /app/frontend/src/pages/MeetingDetailPage.js (handleInviteByEmail)
- Backend API: POST /api/auth/register
- Field name corrected: full_name → name

--------------------------------------------------------------------------------
5. PARTICIPANTS CAN ADD PATIENTS TO MEETINGS
--------------------------------------------------------------------------------

Description: Both organizers and participants can add existing patients to 
meetings, with explicit permission validation.

Features:
- Backend permission checks for both organizer and participant
- "Add Patient" button visible to all meeting members
- Select from existing patients in database
- Cannot add duplicate patients
- Tracking of who added each patient (added_by field)
- Clear error messages for unauthorized access

Location: Meeting Detail Page → Overview Tab → Patients Section

Permissions:
- Organizers: Can add patients
- Participants: Can add patients
- Non-members: Cannot add patients (403 error)

Technical Details:
- Frontend: /app/frontend/src/pages/MeetingDetailPage.js
- Backend API: POST /api/meetings/{meeting_id}/patients
- Backend API: POST /api/meetings/{meeting_id}/agenda
- Database: meeting_patients collection

--------------------------------------------------------------------------------
6. CREATE NEW PATIENT FROM MEETING DIALOG
--------------------------------------------------------------------------------

Description: Create new patients directly from the "Add Patients to Meeting" 
dialog without leaving the meeting page.

Features:
- Two-tab interface: "Existing Patients" | "Create New Patient"
- Required fields: First Name, Last Name, MRN, Date of Birth
- Optional fields: Diagnosis, Phone Number
- Date of birth validation (cannot be future date)
- MRN uniqueness validation
- Patient automatically added to meeting after creation
- Success message with patient details
- Form resets after successful creation

Location: Meeting Detail Page → Add Patient Dialog → Create New Patient Tab

Permissions:
- Organizers: Can create and add patients
- Participants: Can create and add patients

Technical Details:
- Frontend: /app/frontend/src/pages/MeetingDetailPage.js
  - State: newPatient, creatingPatient, patientTab
  - Handler: handleCreateAndAddPatient()
- Backend API: POST /api/patients (creates patient)
- Backend API: POST /api/meetings/{meeting_id}/patients (adds to meeting)

--------------------------------------------------------------------------------
9. CHANGE PARTICIPANT ROLE FEATURE (ORGANIZER ONLY)
--------------------------------------------------------------------------------

Description: Organizers and admins can change any participant's role between 
Doctor, Nurse, Admin, and Organizer.

Features:
- Small edit icon (pencil) next to each role badge
- Inline role editor with dropdown
- Four role options: Doctor | Nurse | Admin | Organizer
- Auto-saves on selection (no separate save button)
- Cancel button (X) to exit without changes
- Role badge color updates immediately
- Success message confirmation
- Backend permission validation
- Audit trail (tracks who made changes)

Location: Participants Page → Participant Card → Edit Icon (✏️)

Permissions:
- Organizers: Can change roles ✅
- Admins: Can change roles ✅
- Doctors: Cannot change roles (edit icon hidden) ❌
- Nurses: Cannot change roles (edit icon hidden) ❌

How to Use:
1. Find participant card
2. Click small pencil/edit icon next to role badge
3. Dropdown appears with role options
4. Select new role
5. Auto-saves and badge updates
6. Click X to cancel without saving

Technical Details:
- Frontend: /app/frontend/src/pages/ParticipantsPage.js
  - State: editingRole, updatingRole
  - Handler: handleUpdateRole(participantId, newRole)
  - Permission check: isOrganizer
- Backend API: PUT /api/users/{user_id}/role (NEW ENDPOINT)
  - Permission check: user.role in ['organizer', 'admin']
  - Validates role is one of: doctor, nurse, admin, organizer
  - Returns 403 if unauthorized
- Database: users collection (role field updated with timestamp)

Role Badge Colors:
- Doctor: Blue (bg-blue-100)
- Nurse: Green (bg-green-100)
- Admin: Purple (bg-purple-100)
- Organizer: Purple (bg-purple-100)

11. EMAIL INTEGRATION SYSTEM ✨ NEW
--------------------------------------------------------------------------------

Description: Comprehensive email notification system that automatically sends 
meeting invitations, response alerts, reminders, and daily digests.

Features:

**1. Meeting Invite Notifications 📩**
- Sent automatically when a meeting is created
- Recipients: All invited participants (except organizer)
- Content: Meeting details, description, quick action buttons (Accept/Decline)
- HTML formatted with responsive design

**2. Response Change Alerts 🔔**
- Sent when a participant accepts, declines, or responds "maybe"
- Recipient: Meeting organizer
- Content: Participant name, response status, meeting details
- Real-time notifications

**3. Meeting Reminders ⏰**
- 24-hour reminder: Sent 1 day before the meeting
- 1-hour reminder: Sent 1 hour before the meeting
- Recipients: All participants who accepted the meeting
- Content: Meeting reminder with countdown, location, join link

**4. Daily Digest 📅**
- Sent every morning at 8:00 AM
- Recipients: All users with upcoming meetings
- Content: Today's meetings, this week's schedule, quick dashboard link
- Personalized for each user

Location: Backend automated system

Permissions:
- All users: Receive relevant email notifications
- Organizers: Receive response alerts
- Participants: Receive invites and reminders

Technical Details:

Backend Files:
- /app/backend/utils/email.py - Email sending utilities

Configuration:

Email Features:

Integration Points:

Email Templates:

                        DATABASE SCHEMA CHANGES
================================================================================

MEETINGS COLLECTION:
- Added: completed_at (String, ISO timestamp)
  Purpose: Track when meeting was marked as completed for 7-day rule

MEETING_PARTICIPANTS COLLECTION:
- Added: response_status (String: 'pending', 'accepted', 'maybe', 'declined')
  Purpose: Track participant's response to meeting invitation
- Added: added_by (String, user ID)
  Purpose: Track who added this participant to the meeting

AGENDA_ITEMS COLLECTION:
- Added: added_by (String, user ID)
  Purpose: Track who created this agenda item
- Added: last_updated_by (String, user ID)
  Purpose: Track who last updated the treatment plan

MEETING_PATIENTS COLLECTION:
- Added: added_by (String, user ID)
  Purpose: Track who added this patient to the meeting

USERS COLLECTION:
- Added: updated_at (String, ISO timestamp)
  Purpose: Track when user details were last modified (especially role changes)

================================================================================
                        API ENDPOINTS ADDED/MODIFIED
================================================================================

NEW ENDPOINTS:
1. PUT /api/meetings/{meeting_id}/participants/{user_id}/response
   Description: Update participant's meeting response status
   Body: { "response_status": "accepted|maybe|declined" }
   Returns: Updated participant object

2. GET /api/health
   Description: Health check endpoint for monitoring and Docker
   Returns: { "status": "healthy", "database": "connected", "timestamp": "..." }

3. PUT /api/users/{user_id}/role
   Description: Update user's role (Organizer/Admin only)
   Body: { "role": "doctor|nurse|admin|organizer" }
   Permissions: Only organizer or admin can call
   Returns: Updated user object

MODIFIED ENDPOINTS:
1. POST /api/meetings/{meeting_id}/participants
   - Added permission check for both organizer and participants
   - Added added_by field tracking
   - Improved error messages

2. POST /api/meetings/{meeting_id}/patients
   - Added permission check for both organizer and participants
   - Added added_by field tracking
   - Returns 403 if unauthorized

3. POST /api/meetings/{meeting_id}/agenda
   - Added permission check for both organizer and participants
   - Added added_by field tracking
   - Returns 403 if unauthorized

4. PUT /api/meetings/{meeting_id}/agenda/{item_id}/treatment-plan

   - Returns 403 with detailed message if expired
