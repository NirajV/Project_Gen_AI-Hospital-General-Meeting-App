# BioMedMeet — Comprehensive Technical Recreation Prompt

> **Purpose**: A single, exhaustive, copy-paste-able prompt that lets any
> capable engineer (or AI coding agent) recreate the BioMedMeet application
> end-to-end — components, features, data models, integrations, Docker
> deployment, and the public marketing site — without referring to any
> other document.

---

## 0. Standing instructions

When following this prompt, the implementer **MUST**:

1. Read every section before writing code. Skim, then build.
2. Never invent functionality not listed here. If something is unclear, mark
   it `TODO` and surface the question; do not guess clinical workflow.
3. Treat patient data with care. Every endpoint that touches patient records
   must require a valid JWT (`Depends(get_current_user)`).
4. Use the exact tech stack listed in §1. Substitutions (Next.js, Prisma,
   Postgres) are not equivalent.
5. Keep frontend pages small (`< 1000 lines` each). Dialog and tab components
   live in `frontend/src/components/meeting/` and
   `frontend/src/components/meeting/dialogs/`.
6. Backend routes prefix `/api`. Frontend calls `/api/*` same-origin via
   nginx, so the React bundle is host-agnostic (works over LAN IP,
   Tailscale, Cloudflare tunnel without rebuild).
7. Use `datetime.now(timezone.utc)` everywhere. No naive datetimes.
8. Exclude `_id` from every MongoDB read (`{"_id": 0}` projection).
9. Email sending is best-effort: API calls succeed even when SMTP fails;
   log a warning, never raise.
10. All env vars come from `.env` (root, next to `docker-compose.mongodb.yml`)
    and are referenced via `${VAR}` in compose. Backend `.env` is for local
    development only and is **not** read by Docker.

## 5. Backend — required endpoints

All under `/api`. JWT required unless noted (`PUBLIC`).

### Auth
- `POST /auth/register` — email + password + name → returns `{access_token, user}`
- `POST /auth/login` — returns `{access_token, user, requires_password_change}`
- `GET  /auth/me` — current user
- `POST /auth/change-password` — current + new
- `POST /auth/google/...` — Emergent Google OAuth flow (optional)

### Users
- `GET /users` — list (filter by org)
- `GET /users/{id}`
- `PUT /users/{id}` — profile + regional + holiday updates

### Meetings
- `GET /meetings` — list (filter by status, date range)
- `POST /meetings` — create + auto-Teams + invite emails + .ics
- `GET /meetings/{id}` — full detail (includes participants + patients + agenda + files + decisions)
- `PUT /meetings/{id}` — update; if `meeting_date` or `start_time` changed → Teams `update_online_meeting` + reschedule emails (full-invite template, `is_update=True`)
- `DELETE /meetings/{id}` — soft cancel
- `POST /meetings/{id}/respond` — body `{response_status: accepted|declined}`
- `POST /meetings/generate-summary-pdf` → returns binary PDF

### Participants
- `POST /meetings/{id}/participants` — body: `user_id` (existing) OR `{email, name, …, invite_new: true}`
- `DELETE /meetings/{id}/participants/{user_id}`
- For invite-by-email: creates user with random temp password, `must_change_password=true`, sends `account_setup.html` email with **platform_url = `{frontend_url}/home/`** (per Bug Fix 1)

### Patients
- `GET /patients`, `POST /patients`, `GET /patients/{id}`, `PUT /patients/{id}`
- `POST /meetings/{id}/patients` — body `{patient_ids: [...]}` (organisers auto-approve; non-organisers need approval)
- `DELETE /meetings/{id}/patients/{patient_id}`
- `POST /meetings/{id}/patients/{patient_id}/approve`

### Agenda
- `POST /meetings/{id}/agenda` — fields per `agenda_items` schema
- `PUT /agenda/{id}` (treatment plan, status)
- `DELETE /agenda/{id}`

### Files
- `POST /meetings/{id}/upload-file` — multipart, fields: `file`, `file_type`, optional `patient_id`
- `GET /files/{file_id}` — **requires Bearer** (Bug Fix 4 — frontend uses `downloadFile()` helper with Blob URL)
- `DELETE /files/{file_id}`

### Decisions
- `POST /meetings/{id}/decisions`
- `PUT /decisions/{id}`, `DELETE /decisions/{id}`

### Teams (Microsoft Graph)
- `POST /teams/generate-link` — standalone Teams link for an existing meeting

### Marketing & Feedback
- `POST /contact` — **PUBLIC** lead capture (demo + cheat-sheet forms post here). Stores in `marketing_leads` collection and emails `OWNER_EMAIL`.
- `POST /feedback` — authenticated user feedback → emails owner

### Health
- `GET /health` — `{status: "ok"}`

### File download flow (Bug Fix 4 — critical)
```javascript
// lib/api.js
export const downloadFile = async (fileId, fallbackName = 'download') => {
    const response = await api.get(`/files/${fileId}`, { responseType: 'blob' });
    const disp = response.headers['content-disposition'] || '';
    const match = disp.match(/filename\*?=(?:UTF-8'')?["']?([^;"']+)["']?/i);
    const filename = match ? decodeURIComponent(match[1]) : fallbackName;
    const blobUrl = window.URL.createObjectURL(new Blob([response.data], {
        type: response.headers['content-type'] || 'application/octet-stream',
    }));
    const a = document.createElement('a');
    a.href = blobUrl; a.download = filename;
    document.body.appendChild(a); a.click(); a.remove();
    setTimeout(() => window.URL.revokeObjectURL(blobUrl), 1500);
};
```
Wired into `FilesTab.js` and `PatientDetailPage.js` Download buttons (onClick, with toast on 401).

## 9. Scheduler (`backend/scheduler.py`)

Single asyncio task started in FastAPI's `startup` event. Two responsibilities:

### A. 1-hour reminder loop
- Poll every `REMINDER_POLL_SECONDS` (default 300)
- For each meeting with `now < meeting_start <= now + 1h` and `reminder_1h_sent != true`:
  - Send `meeting_reminder.html` to each accepted participant
  - Set `reminder_1h_sent: true` (atomic, race-safe)

### B. Auto-complete
- For each meeting with `status in {scheduled, in_progress}` and `now >= end_time + AUTO_COMPLETE_GRACE_MINUTES`:
  - Atomic update: `{$set: {status: "completed", auto_completed: true, auto_completed_reason: "Scheduled end + Xmin grace elapsed", completed_at: now}}`
- Use organiser's timezone via `ZoneInfo` to compute the end-time-in-UTC threshold

Both behaviours are env-gated (`EMAIL_REMINDERS_ENABLED`, `AUTO_COMPLETE_ENABLED`).

RFC-5545 compliant `.ics` produced for every meeting invite. Must include:
- `BEGIN:VCALENDAR` … `PRODID:-//BioMedMeet//Hospital Meeting//EN`
- `VTIMEZONE` block for organiser's IANA zone
- `VEVENT`:
  - `UID:{meeting_id}@biomedmeet`
  - `DTSTART;TZID=...`, `DTEND;TZID=...`
  - `SUMMARY`, `LOCATION`, `DESCRIPTION`
  - `ORGANIZER`, `ATTENDEE`
  - `RRULE:FREQ=WEEKLY|DAILY|MONTHLY|YEARLY[;COUNT=...]` when recurrence chosen

Attached as `application/octet-stream; name="<meeting_title>.ics"` in `send_email`.

## 13. Docker Compose

### `docker-compose.mongodb.yml`
3 services: `mongo`, `backend`, `frontend`. The frontend is a single `nginx:1.27-alpine` container that serves the `yarn build` output AND proxies `/api` → `backend:8001`.

Critical compose snippets:
```yaml
services:
  frontend:
    build:
      context: ./frontend
      args:
        # MUST be empty so the bundle calls same-origin /api/*
        - REACT_APP_BACKEND_URL=
    environment:
      - REACT_APP_BACKEND_URL=
    ports:
      - "3000:80"
  backend:
    environment:
      - MONGO_URL=${MONGO_URL}
      - SMTP_PASSWORD=${SMTP_PASSWORD:-}
      - GRAPH_CLIENT_SECRET=${GRAPH_CLIENT_SECRET:-}
      # …all SMTP_*, GRAPH_*, scheduler vars passed through here
```

### `frontend/nginx.conf`
```nginx
server {
    listen 80;
    location / {
        root /usr/share/nginx/html;
        index index.html;
        try_files $uri $uri/ /index.html;
    }
    location /api/ {
        proxy_pass http://backend:8001;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_cache_bypass $http_upgrade;
    }
}
```

### Deploy
```bash
sudo docker compose -f docker-compose.mongodb.yml up -d --build
```
